# OLOS Object Binding

Status: `draft-v0.1.2`

## 1. Abstract

The OLOS Object Binding defines how OLOS media objects are published to object-like storage providers.

The default v0.1.2 model is **single-bucket direct object publication**:

```text
publisher uploads directly to a coordinator-issued media object key
coordinator verifies that exact object exists
coordinator commits the upload slot
coordinator adds object to the CommittedWindow
manifest gateway emits the object URL from committed state
```

Private upload/public promotion remains a valid optional hardening mode, but is not required for OLOS-Object conformance.

## 2. Object-store assumptions

An OLOS-compatible object binding SHOULD provide:

```text
create object if absent
read object
head object
resolve public delivery URL
grant scoped upload access
declare provider capabilities
```

A provider MAY also provide:

```text
copy/promote object
read gating
provider events
cache control hooks
```

A provider MAY be object storage, an active origin, or a hybrid system if it satisfies the OLOS publication contract.

## 3. Publication models

### 3.1 Direct object publication

This is the default object-store mode.

```text
publisher receives pre-signed PUT for exact object key
publisher uploads object
coordinator observes object existence
coordinator commits slot
coordinator adds object to the CommittedWindow
manifest emits the same object key as the delivery object
```

Example key:

```text
media/{tenant_id}/{session_id}/e{epoch}/{rendition_id}/s{msn}/p{part}-{slot_nonce}.m4s
```

In public single-bucket deployments, direct object publication provides **manifest-gated publication**, not unreadability before commit. A publisher that knows the object key MAY be able to share the direct object URL before the object is emitted in the official playlist.

### 3.2 Direct object publication with read gate

A deployment MAY keep the single bucket and exact object key model while adding a read gate.

Examples:

```text
CDN token authentication
Worker-mediated read authorisation
signed playback URLs
provider-specific access controls
```

This model provides stronger protection against direct access to uncommitted uploads, at the cost of additional delivery-path complexity.

### 3.3 Private upload + public promotion

A deployment MAY upload to a private key and promote/copy to a public key after commit.

```text
publisher uploads private candidate object
coordinator verifies object
coordinator copies/promotes object to public key
coordinator adds public key to the CommittedWindow
manifest emits public key
```

This is a hardening option. It is not the default OLOS v0.1.2 model.

## 4. Abstract operations

### 4.1 CreateUploadGrant

Creates a short-lived upload grant for one slot.

Input:

```json
{
  "slotId": "slot_01JZ",
  "method": "PUT",
  "objectKey": "media/tenant_acme/sess_01JZLIVE/e1/v1080/s3812/p3-slot_01JZ.m4s",
  "contentType": "video/mp4",
  "maxBytes": 524288,
  "expiresAt": "2026-06-08T12:00:05Z",
  "createIfAbsent": true
}
```

Output:

```json
{
  "method": "PUT",
  "url": "https://storage-provider.example/...",
  "expiresAt": "2026-06-08T12:00:05Z",
  "requiredHeaders": {
    "Content-Type": "video/mp4",
    "If-None-Match": "*",
    "x-olos-slot-id": "slot_01JZ"
  }
}
```

### 4.2 HeadObject

Returns metadata for an object without reading the object body.

Output:

```json
{
  "objectKey": "media/tenant_acme/sess_01JZLIVE/e1/v1080/s3812/p3-slot_01JZ.m4s",
  "exists": true,
  "size": 312500,
  "etag": "\"9b2cf535f27731c974343645a3985328\"",
  "contentType": "video/mp4",
  "lastModified": "2026-06-08T12:00:01.750Z"
}
```

### 4.3 ResolveDeliveryUrl

Returns the URL used by playback clients.

Input:

```json
{
  "objectKey": "media/tenant_acme/sess_01JZLIVE/e1/v1080/s3812/p3-slot_01JZ.m4s",
  "pathwayId": "r2-primary"
}
```

Output:

```json
{
  "url": "https://media.example.com/media/tenant_acme/sess_01JZLIVE/e1/v1080/s3812/p3-slot_01JZ.m4s"
}
```

### 4.4 CopyObjectToPublicPath

Optional operation for deployments using private upload/public promotion.

## 5. Object layout

The canonical direct-public layout is:

```text
media/{tenant_id}/{session_id}/e{epoch}/{rendition_id}/init-{slot_nonce}.mp4
media/{tenant_id}/{session_id}/e{epoch}/{rendition_id}/s{msn}/p{part}-{slot_nonce}.m4s
media/{tenant_id}/{session_id}/e{epoch}/{rendition_id}/s{msn}/segment-{slot_nonce}.m4s
```

Rules:

```text
object keys MUST be generated by the coordinator
publisher-supplied object keys MUST be rejected
object keys MUST include enough entropy to make future object URLs non-obvious
committed media object keys MUST be immutable
announced media object keys MUST NOT be overwritten
object keys MUST NOT be reused within an epoch
object keys MUST NOT contain traversal sequences
object keys MUST NOT contain absolute URLs
object identity MUST NOT depend on query strings
```

## 6. Commit checks

Before committing a slot, the coordinator MUST check:

```text
slot exists
slot has not expired, unless late tolerance is explicitly configured
object exists
object key exactly matches the slot object key
object size is within slot limits
object content type matches the slot metadata when available
observed ETag/checksum is consistent
slot has not been committed with a different object
session is still active
publisher remains authorised
quota policy allows publication
```

The coordinator MAY skip byte-level media parsing in no-scan mode.

## 7. Event model

A storage provider may notify the coordinator of object changes.

Normalised event:

```json
{
  "eventId": "evt_01JZ",
  "eventType": "object.created",
  "providerId": "r2-primary",
  "bucket": "olos-media",
  "objectKey": "media/tenant_acme/sess_01JZLIVE/e1/v1080/s3812/p3-slot_01JZ.m4s",
  "size": 312500,
  "etag": "\"9b2cf535f27731c974343645a3985328\"",
  "eventTime": "2026-06-08T12:00:01.750Z"
}
```

Event processing MUST be idempotent.

A provider event MUST NOT by itself advance the cursor. It may only cause an upload slot to be observed and evaluated for commit.

## 8. Cache requirements

Object-store mode MUST be cache-safe.

Requirements:

```text
committed media URLs MUST be immutable
object keys SHOULD be unguessable before playlist emission
future media URLs SHOULD NOT be requested before object existence
404 responses for live future paths SHOULD NOT be cached
preload hints SHOULD be disabled by default in direct object-store mode
overwrites MUST NOT be used for live media
deletes MUST NOT be followed by key reuse
```

## 9. Object metadata

OLOS implementations SHOULD attach metadata where providers support it.

Recommended metadata:

```text
x-olos-session-id
x-olos-slot-id
x-olos-epoch
x-olos-rendition-id
x-olos-media-sequence-number
x-olos-part-number
x-olos-kind
```

Metadata is not authoritative. The slot ledger is authoritative.

## 10. Provider bindings

Provider bindings MUST map these abstract operations to concrete storage operations.

Examples:

```text
R2/S3: pre-signed PUT + If-None-Match + HeadObject
GCS: signed upload URL + generation precondition + object metadata lookup
Azure: signed upload + conditional blob creation + blob properties lookup
Self-hosted: HTTP PUT with create-if-absent semantics + HEAD
```

## 11. Security note

Checking existence before playlist inclusion is required, but it is not the same as proving media safety or preventing direct URL sharing.

A direct-public deployment MUST document that:

```text
uncommitted objects may be directly readable if their URLs are known
publication is manifest-gated unless a read gate is added
No-Scan Mode does not prove hostile media bytes are safe
```
